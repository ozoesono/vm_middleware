VM Middleware
Simple Architecture

What
A service that pulls vulnerability findings from Tenable, scores them against our business context, and creates Jira tickets for service teams to fix.
Why
Tenable has the data but no way to route it to teams.
Jira has the workflow but no risk scoring.
Service teams need prioritised, actionable tickets — not raw scan output.
Flow
Tenable Exposure Mgmt  →  Middleware  →  Jira
                            │
                            ▼
                       PostgreSQL
                            │
                            ▼
                       CSV Reports

Pull — daily call to Tenable's Inventory API.
Enrich — add portfolio, service, environment, criticality from tags.
Score — apply risk formula (VPR + asset criticality).
Reconcile — compare with stored findings to detect new, fixed, or recurring.
Route — create, update, close Jira tickets.
Report — generate CSV exports on demand.

Tech
Python + PostgreSQL
AWS Lambda runs the pipeline on a schedule
ECS Fargate hosts the API service
Single source of truth: Tenable's unified Inventory API
Key Decisions
Decision
Rationale
One Tenable API
Inventory aggregates VM, Cloud, WAS, and container findings
Finding ID is the key
Tenable's stable ID drives deduplication and reconciliation
Pull, don't listen
We re-pull each run; no webhooks needed
Tenable's state is truth
Only FIXED status closes tickets; missing findings are flagged, not auto-closed
